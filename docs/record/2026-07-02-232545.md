# 2026-07-02 23:25:45 变更记录

## 为什么变更

复审当前 Aliyun DashScope Realtime 接入和工具配置后，发现 Aliyun 模型名仍复用通用 `MODEL_NAME`，容易与 OpenAI/Gemini 模型覆盖互相污染；Aliyun camera `duration_seconds` 后台视觉序列也没有纳入统一后台任务管理，取消和状态查看不够干净。另一个配置层面的结论是：联网搜索、天气、SMTP、默认收件人属于普通工具参数，应与 `BACKEND_PROVIDER` 解耦。

## 变更内容

- 配置层：新增 `ALIYUN_REALTIME_MODEL`，Aliyun backend 只读取该变量；`MODEL_NAME` 保留给 OpenAI/Gemini。控制台切换 backend 时会清理不属于目标 backend 的旧模型变量，避免跨供应商污染。
- Aliyun 视觉层：将异步 1 FPS camera sequence 接入 `BackgroundToolManager.start_coroutine_tool()`，返回可追踪 `tool_id`；新增系统工具 `cancel_aliyun_camera_sequence`，也可继续用 `task_status` / `task_cancel` 查看和取消。
- Camera 工具：`duration_seconds` 只在 Aliyun backend 下触发连续视觉序列，其他 backend 继续返回单帧图片；保留 120 秒上限。
- 文档：更新 `.env.example`、`README.md`、`README_zh.md`，明确 `ALIYUN_REALTIME_MODEL`、Aliyun 视觉序列取消方式，以及联网搜索/天气/SMTP/默认收件人的 backend 无关属性。
- 测试：补充 Aliyun runtime config、backend 模型名隔离、非 Aliyun camera duration 忽略、Aliyun sequence cancel tool 等覆盖。
- 本地配置：已读取 `.env` 的变量名并只做最小迁移，将 Aliyun 的旧 `MODEL_NAME` 行改为 `ALIYUN_REALTIME_MODEL`；未改动密钥值。`.env` 是本地敏感配置文件，不应提交。
